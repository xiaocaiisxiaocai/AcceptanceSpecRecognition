## Context

当前系统的匹配链路以 Embedding 为主，按 `项目 + 规格` 组合文本计算相似度，并将得分最高的候选作为默认最佳匹配。真实样本文档评估表明，这一方案在以下场景存在不足：

- `Word` 文档中同一项目名下存在多条不同规格
- `Excel` 文档中项目字段高度重复，必须依赖规格与其它结构字段区分
- 数值、单位、范围词、工艺动作等结构化信号对最终正确性有显著影响

另一方面，系统已经具备以下可复用能力：

- 文本预处理
- Embedding 批量生成与缓存
- LLM 复核与建议
- 预览与批量预览接口

因此，本次设计选择在当前链路上增加“多阶段召回重排”，而不是替换现有单阶段方案。

## Goals / Non-Goals

- Goals:
  - 在复杂和高歧义样本上提高最佳匹配排序正确率
  - 保持默认行为与当前版本兼容
  - 把额外成本限制在需要重排的场景中
  - 提供足够的结果解释信息，便于用户理解和排查
- Non-Goals:
  - 不在本次变更中重做全文档导入模型
  - 不把 LLM 变成必选主排序器
  - 不在本次变更中修改 `AcceptanceSpec` 主表结构
  - 不默认对所有请求启用多阶段链路

## Decisions

### Decision: 新增匹配策略与高级参数，但默认维持单阶段

为匹配配置增加以下参数：

- `MatchingStrategy = SingleStage | MultiStage`
- `RecallTopK`
- `RerankMode = RulesOnly | RulesPlusLlm`
- `OnlyForAmbiguousMatches`
- `AmbiguityMargin`

当新参数未传时，系统应继续按当前逻辑执行单阶段匹配，确保兼容现有前端与调用方。

### Decision: `MinScoreThreshold` 在多阶段中作为第一阶段候选准入阈值

为了保持现有 `MinScoreThreshold` 的语义稳定，本次约定：

- `SingleStage`：继续按当前行为，低于 `MinScoreThreshold` 的候选直接过滤
- `MultiStage`：先按 Embedding 得分过滤，仅保留 `EmbeddingScore >= MinScoreThreshold` 的候选进入召回集合
- 如果没有候选满足该阈值，则直接返回无匹配结果
- 第二阶段重排仅对已通过第一阶段阈值的候选重新排序，不负责“救回”低于阈值的候选

这样做的原因：

- 不破坏现有配置项的业务含义
- 让多阶段仍然建立在“先有基础语义相关性”之上
- 避免引入一个新的、与现有阈值含义冲突的过滤规则

### Decision: 多阶段模式下先召回 TopK，再做规则重排

多阶段模式的核心流程：

1. 用 Embedding 召回 `TopK` 候选
2. 用规则重排对候选重新评分
3. 选择重排后最佳结果作为默认匹配
4. 如候选仍高度接近，再按配置触发 LLM 复核

这样做的原因：

- 先保证正确答案尽量进入候选集
- 再利用结构化信号提升排序正确率
- 避免在所有样本上无差别调用 LLM

### Decision: 第二阶段优先使用轻量规则重排

第一版重排规则优先考虑以下信号：

- 项目文本完全一致或归一化后高度一致
- 规格中的数值、单位、范围词一致
- 关键词重合度
- 否定词、方向词、工艺动作冲突
- 结构性差异（例如项目相同但规格明显分叉）

理由：

- 规则重排可解释性更强
- 成本和时延可控
- 更适合作为第一版稳定落地

### Decision: 仅对高歧义样本触发 LLM 复核

LLM 不参与所有样本的默认排序，只在以下条件满足时参与：

- 启用了 `RulesPlusLlm`
- 候选分差落在歧义区间内
- 或系统判定当前结果属于高风险误判样本

这样可以把 LLM 的角色限定为“歧义裁判”，而不是“全量排序器”。

### Decision: 第一版歧义判定采用“重排后前两名分差阈值”

第一版歧义规则定义为：

- 多阶段重排完成后，若至少存在两条候选
- 且 `Top1FinalScore - Top2FinalScore <= AmbiguityMargin`
- 则该样本被判定为“高歧义样本”

补充约定：

- 若只有 0 或 1 条候选通过第一阶段阈值，则不判定为歧义
- 第一版以“分差阈值”作为唯一规范判定标准，避免前后端、测试和实现出现多套口径

这样做的原因：

- 规则简单、可测试、可解释
- 便于前端直接展示和调参
- 先建立统一口径，再根据后续评估结果决定是否引入更复杂的组合判定

### Decision: 执行填充必须复用预览确认结果

为了避免用户在预览中看到的是 A，执行填充时又被系统静默替换成 B，本次约定：

- 执行填充必须使用用户在预览阶段确认后的候选结果或建议结果
- 若请求中已显式携带用户确认的 `SpecId / SelectedSpecId / LLM 建议内容`，执行阶段不得重新按当前策略自动替换为新的默认匹配
- 只有当用户重新发起预览并重新确认后，新的匹配结果才可进入执行阶段

这样做的原因：

- 保证用户感知的一致性
- 降低多阶段策略带来的“预览与最终结果不一致”风险
- 让测试可以明确验证“预览确认结果就是最终填充结果”

### Decision: 预览结果需要输出阶段性元数据

为了让前端能够解释匹配结果，预览结果需要携带以下额外信息：

- 当前使用的策略
- 是否走了多阶段
- 召回候选数量
- 是否为歧义样本
- 重排得分或重排原因摘要
- 是否触发了 LLM 复核

## Alternatives Considered

### Alternative: 保持当前单阶段方案，仅调阈值

优点：

- 实现成本最低
- 对现有代码侵入最小

缺点：

- 无法解决“正确答案被召回但没排第 1”的核心问题
- 对高重复项目、近似规格场景帮助有限

### Alternative: 直接使用 LLM 做全量重排

优点：

- 语义判断能力更强

缺点：

- 成本更高
- 时延更高
- 解释性更差
- 对所有请求全量启用风险较大

本次不采用该方案作为首选。

## Risks / Trade-offs

- 增加匹配链路复杂度，需要更清晰的测试和结果解释
- 如果规则设计不当，可能对某些模板过拟合
- 多阶段模式会增加预览耗时，需要控制默认触发条件
- 如果 LLM 介入条件过宽，可能导致时延和成本失控

Mitigation:

- 默认保留单阶段
- 第一版优先 `RulesOnly`
- 仅对歧义样本开放 LLM
- 通过真实样本建立基线并持续回归验证

## Migration Plan

1. 扩展请求配置模型，给所有新参数提供默认值
2. 在 Core 层增加多阶段分支，但保留当前单阶段实现
3. 更新 API DTO 和前端界面，使用户可以显式选择策略
4. 为单阶段兼容、多阶段排序、歧义触发补充测试
5. 基于真实样本文档建立准确率基线，并逐步校准默认参数

## Open Questions

- `RecallTopK` 的默认值是否固定为 `5`，还是按文档类型区分
- Excel 的 `验收方式`、`评议大纲` 等字段是否纳入第二阶段特征输入
- 第一版是否需要在预览弹窗中展示全部候选明细，还是只展示最终结果和摘要
