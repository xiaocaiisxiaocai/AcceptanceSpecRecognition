## Context
当前验收规格页主要支持按客户/机型/制程分组后进行关键词搜索。这个模型适合“已知文本片段”的查找，但不适合“语义相近、措辞不同”的历史规格复用。  
项目已经具备 Embedding、匹配排序与缓存能力，因此新增“验收规格 AI 语义搜索”更适合复用现有向量链路，而不是另起一套文本算法。

## Goals / Non-Goals
- Goals:
  - 支持单条和多条输入的语义搜索
  - 按当前规格分组范围与数据权限返回结果
  - 返回相似度分数并支持逐条编辑结果
  - 不新增数据库表，尽量复用现有 Embedding 缓存
- Non-Goals:
  - 不支持文件上传式批量搜索
  - 不支持对搜索结果做自动批量改写
  - 不引入 LLM 解释、摘要或改写建议
  - 不改变现有普通关键词搜索入口

## Decisions

- Decision: 在“验收规格”页面增加独立 `AI搜索` 面板，而不是替换现有搜索框
  - Why: 关键词检索仍有价值，直接替换会打断原有使用习惯。
  - Alternatives considered:
    - 复用现有搜索框加一个“AI模式”开关：交互耦合过重，批量输入体验差。
    - 单独做新页面：链路过长，脱离当前分组上下文。

- Decision: 批量输入采用“多行文本”，每行独立搜索并单独分组展示结果
  - Why: 最符合用户“可以批量搜，然后分别处理每一条结果”的诉求。
  - Alternatives considered:
    - 逗号分隔：不适合长文本规格。
    - Excel 上传：实现复杂度高，超出第一版必要范围。

- Decision: 语义检索基于组合文本 `项目 + 规格 + 验收标准 + 备注`
  - Why: 单看规格内容会漏掉项目语义，单看项目+规格又会错过验收标准与备注中的关键语义。
  - Alternatives considered:
    - 仅项目+规格：召回不够完整。
    - 仅规格内容：跨项目误召回风险更高。

- Decision: 搜索结果编辑复用现有“验收规格编辑弹窗”
  - Why: 降低前端改造成本，保持字段校验与保存链路一致。
  - Alternatives considered:
    - 结果表格内联编辑：实现快，但复杂字段内容体验较差。
    - 新建专用编辑页：流程过重。

- Decision: 第一版返回 TopN 结果并允许设置最小分数，但不支持分页深翻
  - Why: 用户目标是快速找近似项并修改，TopN 足够支撑人工处理。
  - Alternatives considered:
    - 无限滚动或分页：增加接口与前端复杂度，第一版收益有限。

- Decision: 权限上新增按钮/API 权限，而页面权限继续沿用验收规格页
  - Why: 便于控制“谁可以打开 AI 搜索”，同时不拆页面级访问模型。

## Backend Shape
- 新增接口：`POST /api/specs/semantic-search`
- 请求结构建议：
  - `queries: string[]`
  - `customerId?`
  - `machineModelId?`
  - `processId?`
  - `machineModelIdIsNull?`
  - `processIdIsNull?`
  - `topK`
  - `minScore`
  - `embeddingServiceId?`
- 响应结构建议：
  - `groups: [{ queryIndex, queryText, totalHits, items: [...] }]`
  - 每条结果包含规格主键、客户/机型/制程、项目、规格、验收标准、备注、相似度分数

## Data / Persistence
- 不新增业务表。
- 复用 `AcceptanceSpec -> EmbeddingCaches` 作为候选向量来源。
- 当候选规格缺少对应 Embedding 缓存时，沿用当前匹配链路即时补齐或批量生成。

## UI Shape
- 在验收规格页右侧工具栏增加 `AI搜索` 按钮。
- 打开后显示侧边抽屉或大弹窗：
  - 左侧：多行输入框、TopN、最小分数、执行搜索按钮
  - 右侧：按输入文本分组的结果列表
- 每个结果组内展示：
  - 查询文本
  - 命中数量
  - 每条规格的分数、项目、规格、验收、备注、所属分组
  - `查看` / `编辑` 操作
- 编辑保存成功后：
  - 同步更新当前搜索结果中的对应项
  - 刷新主规格表当前列表

## Risks / Trade-offs
- Embedding 搜索在大数据量下会增加响应时间
  - Mitigation: 限制单次查询条数、限制 TopN、复用缓存、优先批量生成查询向量
- 组合文本召回可能带来部分“验收标准语义相近但项目不相关”的结果
  - Mitigation: 结果展示时保留项目/分组信息，交由人工判断；后续再考虑加权策略
- 逐条编辑会造成用户在大量结果时操作较多
  - Mitigation: 第一版先保证准确与稳定，不提前引入高风险批量改写

## Migration Plan
1. 新增权限种子与前端按钮权限判断
2. 新增后端语义搜索接口与测试
3. 新增前端 AI 搜索面板与逐条编辑回写
4. 冒烟验证后再考虑是否扩展到批量修改

## Open Questions
- 是否需要在结果中额外标识“与当前分组完全一致 / 跨机型 / 跨制程”差异标签
- 后续是否需要支持“把一条输入命中的多条规格加入待处理清单”再集中编辑
